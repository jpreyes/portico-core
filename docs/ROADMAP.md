# Roadmap — portico-core

**English** · [Español](ROADMAP.es.md)

Status: ⬜ pending · 🟡 in progress · ✅ done.

This document summarizes the public status of **portico-core**, the open-source (AGPL-3.0)
edition of PORTICO: pre/post-processor + 3D viewer + JS solver.

---

## v0.1 — first open-source cut

Goal: publish a clean, professional and reusable base.

- ✅ Full de-branding: no institutional brand or third-party logos.
- ✅ Branding as configuration (`branding.default.json`) for white-label.
- ✅ Concrete naming to **grade G** (NCh170:2016), with backward compatibility for the legacy "H".
- ✅ `SolverBackend` interface (`js/solver/backend.js`): seam for pluggable engines (only the JS engine in open).
- ✅ UI/report extension seam (`js/ext/extensions.js`): config sections, badges and flags that upper
  layers register without forking `app.js`. See [`EXTENDING.md`](EXTENDING.md).
- ✅ Professional mode / token out of core (lives in the Pro layer): core is functional without a token.
- ✅ AGPL-3.0 license and open-core-oriented README.
- ✅ Repository without binaries or copyrighted third-party material.
- ✅ **Static wired through the `SolverRegistry`**: `App._solveStaticCases` queries the active backend
  (hook in place; default `'js'` unchanged) → an external backend (Nodex) solves with transparent
  fallback to JS. Post-processing (diagrams/deformed shape) reusable by backends.

## Next milestones

- ✅ **Internationalization (ES/EN)**: source-string i18n engine (`js/i18n/`) with a language selector
  and fallback to Spanish. Covers menus, header, cover page, **properties panel, toolbar (tooltips),
  dialogs and toasts**; the engine also translates attributes (`title`/`aria-label`/`placeholder`) and
  auto-translates the panels that re-render.
- ✅ **Branding loader**: `js/branding.js` reads `branding.default.json` at startup (before the App) and
  fills the UI name/tagline/description/logo via `data-brand` attributes. White-label without forks.
- ✅ **Registry for all JS analyses**: static, modal, spectrum, buckling, P-Δ, nonlinear, pushover,
  modal time-history, stages, moving loads, tendon and form-finding go through the `SolverBackend` with
  fallback to JS.
- ✅ **Verification suite**: 18 documented cases in `docs/verifications` against analytical solutions /
  published references, with figures generated by `tools/` (no dependency on external PDFs).
- ✅ **API documentation** (`docs/api.md`) and minimal integration examples.
- ✅ **English-canonical documentation**: code comments, README, CHANGELOG, CONTRIBUTING, SECURITY and
  CODE_OF_CONDUCT in English (`.md`), with Spanish kept as `*.es.md`.
- ✅ **Community profile + CI**: `.github/` issue/PR templates and a GitHub Actions workflow that runs
  the test battery on every push/PR.
- ⬜ Packaging portico-core as a dependency consumable by derived products.

## Architecture

The pre/post-processor and the JS solver live in this repo. The heavy engine (C++/WASM) does **not**
live here: it plugs in through `SolverBackend` in the Pro products. Architecture details and
engineering conventions are in [`EXTENDING.md`](EXTENDING.md) and at the top of each module.

## How to contribute

- Issues and discussions in the repository.
- Validate any solver change against an analytical case (`test_*.mjs` pattern in the root).
- UI source strings in Spanish; **code comments and commit messages in English**; documentation in
  English (canonical) with Spanish `*.es.md`.
